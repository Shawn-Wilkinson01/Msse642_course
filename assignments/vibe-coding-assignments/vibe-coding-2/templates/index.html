What Is SQL Injection?

SQL injection is an attack where an attacker inserts malicious SQL code into an input field that is then executed by the database. The attack works because the application builds SQL queries by pasting user input directly into a query string, letting the attacker change the structure of the query, not just its data.

SQL injection has topped the OWASP list for over a decade because it is easy to exploit, often devastating in impact, and still shockingly common in production systems.

Why Is It Dangerous?

  • Authentication bypass — log in as any user without a password
  • Data exfiltration — dump entire tables including passwords, emails, PII
  • Privilege escalation — extract admin credentials or session tokens
  • Data modification — UPDATE or DELETE records
  • Server compromise — on some databases, read/write OS files or execute commands

What Attackers Target

  • Login forms (username / password fields)
  • Search bars and filter inputs
  • URL parameters (?id=5, ?user=alice)
  • REST API request bodies (JSON / form data)
  • HTTP headers (User-Agent, Referer, Cookie)

Anywhere user input eventually touches a database is a potential injection point.

How Parameterized Queries Prevent It

A parameterized query separates the SQL structure from the data. The database receives two things: the fixed query template (WHERE username = ?) and the user value as a separate bound parameter.

The database engine then treats the parameter as pure data — it is never interpreted as SQL syntax. Even if the input contains quotes, hyphens, or SQL keywords, they cannot alter the query logic.

Safe pattern (Python sqlite3)

cursor.execute("SELECT … WHERE username = ?", (user_input,))

Real-World Example — Fortra GoAnywhere MFT (2023)

In early 2023, the Cl0p ransomware gang exploited a zero-day SQL injection vulnerability (CVE-2023-0669) in Fortra's GoAnywhere managed file transfer software. The vulnerability allowed unauthenticated remote code execution via a crafted SQL injection payload in the administrative console. Over 130 organizations were compromised within two weeks — including Community Health Systems (1 million patient records), Hitachi Energy, Procter & Gamble, and City of Toronto. The attack succeeded because one injectable endpoint had not been parameterized, giving attackers direct access to data and, via the database's OS-level features, the ability to deploy ransomware payloads.
